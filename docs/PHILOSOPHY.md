# Philosophy

> *"If you're not failing, you're not innovating enough."*
> — Elon Musk, SpaceX

Our engineering culture borrows from SpaceX, with adjustments for a software-only
company. These are the principles we hold and the anti-patterns we reject.

---

## 1. Simplicity is the ultimate sophistication

The best software is the software that is easy to delete. Before adding a feature,
ask: **what is the simplest thing we could build that solves the operator's problem?**

We reject:
- Microservices when a monolith would do
- Event sourcing when CRUD would do
- Kubernetes when a single VM would do
- A custom DSL when JSON would do
- A custom agent framework when a function call would do

We prefer:
- Boring, well-understood technology
- Single-tenant data planes for security
- Explicit over implicit
- Readable over clever

> *"Perfection is achieved not when there is nothing more to add, but when there is
> nothing left to take away."*
> — Antoine de Saint-Exupéry

## 2. The AI shows its work

Every decision the AI agent makes must be:
- **Explainable** — the operator can ask "why?" and get a coherent answer
- **Reversible** — the AI proposes, the operator disposes
- **Auditable** — every proposal is logged with the inputs that produced it

We reject black-box models in critical paths. A neural net can suggest a candidate
maneuver, but the proposal that goes to the operator must be generated by a
reasoning chain they can follow.

We prefer chain-of-thought, explicit scoring, deterministic tie-breaks, and human-
readable summaries over end-to-end learned policies.

## 3. We are vertically integrated

Every layer of the OrbitOps stack lives in this repository:
- Web UI (vanilla JS + Three.js, no React/Vue/etc.)
- Marketing site (same codebase)
- Backend (Node + TypeScript, a single Fastify service)
- Telemetry pipeline (in-process module)
- AI agent (in-process LangGraph reasoning loop)
- Database schema (SQL migrations in repo)
- Deploy config (Cloudflare Worker + Container — `wrangler.toml`, `worker.js`, `backend/Dockerfile`)

We reject the "best tool for the job" trap. Best-of-breed tools compose into a
maintenance nightmare. We are willing to write more code ourselves in exchange for
fewer third-party surprises at 03:00.

## 4. Speed of iteration > perfection of design

We ship the simplest thing that works, then improve it. We measure ourselves on
**time-to-first-paying-customer-feedback**, not on architecture diagrams.

We reject:
- Six-month planning phases
- Big-bang releases
- Feature flags for everything (use them only when you actually need to)
- Specs longer than the code they describe

We prefer:
- Working software in front of users within 14 days of a decision
- One engineer who ships over five who deliberate
- "Done is better than perfect" — every Friday

## 5. Dogfood relentlessly

OrbitOps ships with a **simulated demo constellation** you can run in the browser
right now — the same cockpit, agent console, and audit log a real operator would
use, driven by real orbital math on clearly-labelled simulated telemetry. The
principle: if a workflow isn't good enough for that demo, it isn't good enough to
ship. Nothing here is a mockup we've never exercised.

We reject the "build it and they will come" fallacy. We are our own first user.

## 6. Boring infrastructure, exciting product

| Layer | Choice | Why |
|---|---|---|
| Database | Postgres (pglite in dev, same SQL) | Boring, reliable, JSONB when we need it |
| Backend | Node + TypeScript (Fastify) | Fast iteration, good tooling, ubiquitous |
| Web | Vanilla JS + Three.js | Smaller bundle, fewer abstractions, readable line by line |
| Build | None | Source is what ships. No surprises. |
| Deploy | Cloudflare Worker + Container (one platform) | Static app and Node backend together, no second vendor |
| Secrets | Cloudflare secrets / file-backed (`<NAME>_FILE`) | Never in the repo or a URL |
| Observability | OpenTelemetry (env-gated, no-op by default) | Standard, and free when unused |

A Redis-backed event bus and a managed database come in only if a hosted, multi-instance tier is ever built (see the roadmap) — the single-instance defaults above are deliberate for the open-source build.

Exciting tech is reserved for the parts that actually matter: the AI reasoning
loop, the 3D visualisation, the manoeuvre planner.

## 7. We hire generalists who ship

Every engineer at OrbitOps is expected to:
- Read the marketing copy and tell us when it lies
- Look at the dashboard and tell us when it confuses them
- Read the AI reasoning chains and tell us when they sound wrong
- Operate the demo constellation for at least one shift per quarter

We reject the "frontend vs backend vs ML vs DevOps" siloing. The cost of context-
switching is higher than the cost of learning the adjacent layer.

## 8. We charge for value, not seats

Our pricing is per-satellite and per-decision-augmented, not per-seat. We do not
want to be in the position of encouraging our customer to add operators so we can
charge them more. We want them to add operators because they have more satellites
to operate.

We reject the "platform tax" — charging per integration, per API call over a tier,
per GB of storage. We charge a single transparent rate that scales with the
constellation, not with our customer's success.

## 9. We say no, often

The biggest risk to a small team is scope creep. We have a single sentence that
describes what we are building:

> *OrbitOps is the AI co-pilot that monitors, predicts and proposes actions for
> commercial satellite constellations, with humans always in the loop.*

Anything that does not serve that sentence gets cut. Every Friday we review the
backlog and ask: does this serve the sentence? If no, it goes.

## 10. We will be wrong, publicly

When we ship a bug, we post-mortem it publicly. When we make a wrong product
decision, we say so. When the AI proposes something wrong, we tell the customer
and the regulator. Reputation in this market is built on trust, and trust is built
on admitting fault fast.

We reject the "we'll fix it in the next release" instinct. The next release is
too late for a customer whose satellite is on a bad orbit because of our bug.

---

## Anti-patterns we will reject in code review

- **Premature abstraction** — three is too early to abstract
- **Magic strings** — if a value matters, give it a name
- **Inheritance for code reuse** — composition, always
- **Global state** — everything flows through the store
- **Hidden side effects** — if a function does something, you can see it
- **Commented-out code** — git remembers
- **Dead exports** — if you don't use it, delete it
- **Tests that test the implementation, not the behaviour** — rewrite them

## What we measure

Every sprint, every quarter, every year:
- **Time from incident detection to operator action**
- **Number of AI proposals approved / rejected / modified by humans**
- **Lines of code in the agent loop that an intern can read in one sitting**
- **Customer churn rate**
- **Demo constellation uptime** (we measure ourselves first)

If we ever optimise for vanity metrics (GitHub stars, conference talks), we will
catch it in the next all-hands.
